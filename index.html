01// 觀念篇

什麼是 Vibe Coding

一句話：用「講人話」描述你要的結果，讓 AI 幫你做出來。

過去要讓電腦做事，你得會「電腦的語言」——程式碼、公式、指令。Vibe Coding 把它反過來：你用自然語言說出想要什麼，AI 負責把它變成可運作的東西。你的工作從「會不會寫」，變成「會不會把問題講清楚、並判斷結果好不好」。

名詞由來

「Vibe coding」一詞 2025 年由 AI 科學家 Andrej Karpathy 提出，意思是「跟著感覺走、用對話一步步把東西做出來」。

用 DIS Tech 的語言來比喻

想像你要一片能測某顆 IC 的測試板。傳統做法像是你親自畫每條 trace、排每個 pad；Vibe Coding 則像你身邊多了一位很強的工程助理：

  • 你說：「我要一塊能測這顆晶片的介面板，pitch 大概這樣，先給我一版。」
  • 助理先做一版 → 你說「這裡訊號完整性再好一點」→ 它再改一版。
  • 你專注在「判斷對不對、要什麼」，繁瑣執行交給它。

以前重要的是…

  • 記得語法、指令、函式
  • 會操作特定軟體
  • 從頭手工做每一步

現在重要的是…

  • 把需求＆情境描述清楚
  • 判斷 AI 結果好不好
  • 用對話一次次修到對
重點帶走

Vibe Coding 不是「取代你」，而是把你變成「指揮者」。你越懂自己的工作、越會描述需求，AI 做出來的東西就越好。

02// 觀念篇

認識 AI Agent

會「聊天」的 AI，和會「動手做事」的 AI，差很多。

你大概用過 ChatGPT 這類聊天 AI——你問問題，它回你一段文字。AI Agent（智慧代理）更進一步：它不只回答，還能實際動手——讀你的檔案、建立新檔、上網查資料、執行工具、一步步把任務做完，做錯還會自己修正。

聊天 AI ＝ 顧問

給你口頭建議：「你應該把名單去重、再分類。」——然後你得自己動手。

AI Agent ＝ 實習生

真的坐到你電腦前，把名單去重、分類、產出乾淨表格交給你。你只要驗收。

Agent 是怎麼工作的

步驟它在做什麼白話比喻
理解讀懂任務和你給的資料聽懂你要什麼
規劃把大任務拆成小步驟列出待辦清單
行動讀檔、查資料、計算、產檔動手做事
檢查看結果對不對自己對答案
修正／交付有問題就改，沒問題就交件交給你驗收
所以重點是…

Agent 能「碰到你的東西」（檔案、資料、工具），這是它強大的原因，也是為什麼第 7 章「安全與界線」很重要。

04// 上手篇

第一次上手

先做好心理建設，再走三個步驟，你就上路了。

心理建設

你不需要會寫程式。你需要的能力是把工作講清楚——這件事你本來就在做（交辦同事、寫 email）。把 AI 當成反應很快、但需要你說明清楚的新同事。

上手三步驟

打開工具，告訴它你要什麼

不用想得太完美，先用一句話講出目標。它會問你問題或直接開始做。

看結果，給回饋

結果不完美很正常。直接說：「不錯，但請把日期改成西元、再加一欄『地區』。」

重複到滿意為止

一來一往幾次就會收斂。這個「對話迭代」就是 Vibe Coding 的核心。

新手常見的三個卡點

「怕弄壞東西」

先複製一份檔案、在測試資料夾裡做。原始檔不動就不怕。

「不知道講什麼」

用第 5 章公式，或直接說「我想做 X，你需要我提供什麼？」

「看不懂回覆」

直接回它：「請用白話、像對非技術同事一樣解釋一次。」

05// 上手篇

下指令的心法

同樣的 AI，指令好不好，結果天差地別。記住一個萬用公式。

好指令的四要素

要素意思例子
情境 Context你是誰、背景是什麼「我是 DIS Tech 業務，有一份展會客戶名單。」
任務 Task你要它做什麼「幫我去重、補齊公司欄位、依地區分類。」
格式 Format輸出要長怎樣「輸出表格，欄位：公司、聯絡人、地區、產業。」
範例 Example給個樣板（可選）「像這樣：台積電 ｜ 王小明 ｜ 台灣 ｜ 晶圓代工」

爛指令 vs 好指令

✕ 太模糊
幫我整理一下這個名單。
# AI 不知道「整理」是去重？排序？分類？格式也沒講。
✓ 清楚可執行
我是 DIS Tech 業務，附上一份展會客戶名單（CSV）。
請幫我：
1) 移除重複的公司
2) 統一公司名稱寫法（全形/半形、簡稱補全）
3) 新增一欄「地區」（台灣 / 日本 / 美國 / 其他）
輸出成乾淨表格，最後用 3 句話摘要這份名單的重點。

三個讓結果更好的小技巧

  • 讓它先說計畫再動手：加一句「先告訴我你打算怎麼做，我確認後再開始」。
  • 給它素材：把檔案、截圖、範例直接給它，比口頭描述準很多。
  • 用回饋迭代：「方向對，但第 2 點請改成……」——一次推進一點。
懶人版口訣

「我是誰 → 我要什麼 → 給我什麼格式」。記住這三句，八成的指令就夠用了。

盤點重複性工作

很多人以為這些工具只給工程師用。其實它最擅長處理大量瑣碎、有規則、要花時間的文書與資料工作——而這正是各部門每天都在做的。先對照下表，找出你的「可外包清單」。

07// 實戰篇 · 練習 01

名片與客戶資料

練習 1 · 名片與客戶資料

把參展回來的一疊名片、加上各來源的潛在客戶，變成乾淨可用的清單與跟進名單。

難度：入門約 15 分鐘業務 / FAE / 行政

下載範例檔 · 01_業務_客戶名單

情境

展會結束，你手上有一疊名片（用手機拍照／掃描），再加上 email 簽名、報名表收來的潛在客戶。問題是：寫法亂、有重複、有缺漏。今天用 AI 把它整理乾淨、並標出該優先跟進誰。

① 名片 → 表格（用看得懂圖片的 AI）

把名片拍成照片，丟給能讀圖的 AI（多模態），請它辨識成表格。指令範例：

＋ 附上名片照片
附上幾張名片照片。請幫我辨識並整理成表格，欄位：
公司、姓名、職稱、email、電話、地址、備註。
看不清楚或不確定的欄位請留空、不要猜。
最後把無法判讀的名片列出來提醒我。

③ 把這段指令貼給 AI

給 AI 的指令
我是 DIS Tech 業務。附上一份展會客戶名單（CSV，很亂）。
請幫我：
1) 合併重複客戶（同公司中英文、或同 email 視為同一筆）
2) 統一公司名稱，保留「中文名 + 英文／代號」
3) 補一欄「地區」：台灣 / 韓國 / 美國 / 其他
4) 補一欄「需求」：從備註判斷（probe card、FT board、封測…），判斷不出留空
5) 補一欄「優先級」：依是否大客戶、需求是否明確，標 高/中/低
email、電話清乾淨；缺漏留空不要猜。

輸出：(A) 整理後的乾淨表格　(B) 3～5 句重點摘要（共幾家、地區分布、優先跟進誰）
開始前先用一句話說你的整理計畫，我確認後再做。
▶ 動畫示範客戶名單去重：10 筆亂名單 → 7 家乾淨清單 + 摘要

④ 進階：把成果變更好用

迭代追問
很好。再幫我：
(1) 依「需求類型」分群，方便分配給對應 FAE
(2) 幫每一家草擬一句客製的開場跟進訊息
(3) 存成我能用 Excel 打開的檔案
學到的觀念

你沒寫任何程式，只是「清楚描述任務 + 指定格式 + 迭代追問」。同手法可套用到供應商清單、報名表、待辦彙整。

08// 實戰篇 · 練習 02

練習 2 · 競品與產品差異

研究 probe card / DIB 市場對手，產出比較表，並整理出「我方的產品區別性」。

難度：入門～進階約 20 分鐘業務 / PM / 策略

下載範例檔 · 10_競品分析_範本

情境

DIS Tech 隸屬 Technoprobe 集團，做晶片測試介面（probe card、DIB／PIB、FT board）。主管想要一份「競爭對手快照 + 我方差異化賣點」，方便業務對話與市場判斷。我們讓 AI 來打底。

① 要比較的對手（參考清單）

FormFactor

美國，高效能 probe card 領導廠之一。

MJC（日本麥克隆尼）

Micronics Japan，vertical／MEMS 探卡。

Technoprobe

義大利集團（DIS Tech 母集團）。

Accuprobe

高精度探針元件與 probe card。

Wentworth Labs

cantilever／vertical／MEMS 探卡。

其他

JEM、Tplus… 可視市場再加。

② 競品分析指令（建議用能上網的 agent）

給 AI 的指令
我在半導體測試介面公司（probe card / DIB / FT board）負責市場研究。
請做一份「競爭對手快照」，比較：
FormFactor、Micronics Japan (MJC)、Technoprobe、Accuprobe、Wentworth Labs。
每家整理：總部地區、主力產品線、技術特色（MEMS / vertical / cantilever）、
鎖定客群或應用、公開可見的優勢與弱點。

輸出：(A) 一張橫向比較表　(B) 每家 2～3 句重點短評　(C) 對我方業務最有用的 3 個觀察
重要：每項資訊標註「有查到來源」或「推測」，沒把握請明說，不要編造數字。
先給我研究計畫，我確認後再開始。

③ 產品區別性（我方差異化）

競品表出來後，接著請 AI 幫你把「我方 vs 對手」的差異整理成業務能用的賣點：

迭代追問
根據上面分析，幫我整理「我方（Technoprobe 集團的 DIS Tech）的產品區別性」：
(1) 我方 vs 對手 的差異化重點（如：turn-key 整合服務、在地設計支援、
    集團垂直整合、交期與量產 readiness、總體成本 TCO）
(2) 做成一張「差異點 → 對客戶的好處」對照表
(3) 業務遇到客戶拿對手比價時，可用的 3 句差異化說法
(4) 一頁式 SWOT 重點
務必人工查核

市場數字、規格、份額這類資訊，AI 可能「講得很自信卻是錯的」。對外使用前，務必由懂市場的同事核對來源。AI 負責打底，你負責定稿。

練習 3 · 用 AI 做簡報

練習 4 · NotebookLM

把你的規格、報告、教育文件，變成「只根據這些資料、附出處」的問答助理。

難度：入門約 15 分鐘工程 / 新人訓練 / 業務

下載範例文件 · 11_NotebookLM_產品知識庫範例

它跟一般聊天 AI 差在哪？

NotebookLM 是 Google 的研究助理，最大特點是：它只根據你上傳的資料回答，而且每個答案都會標出「出自哪份文件、哪一段」。這大幅降低「幻覺」，答案可追溯——很適合拿來做內部知識庫。

上傳來源

PDF、規格、datasheet、簡報、會議記錄、網頁。

提問附出處

「我們的探卡在高頻測試的優勢？」答案會附引用段落。

自動生成

摘要、FAQ、學習指南，甚至 Podcast 式語音導讀。

DIS Tech 可以這樣用

建立 Notebook、上傳來源

把產品規格、技術白皮書、內部教育文件、常見問答丟進去。

讓新人用「問」的方式上手

新進工程師問：「DIB 和 probe card 差在哪？我們的 turn-key 流程是什麼？」即時得到有出處的答案。

幫業務／FAE 快速備課

問：「客戶問高頻應用，我們有哪些對應方案？」→ 產出重點＋出處，準備拜訪更快。

一鍵生成教育素材

讓它把一份厚規格生成「重點摘要」「FAQ」或「語音導讀」，當作訓練教材。

上傳前先確認

上傳的是公司內部文件，請先確認公司資料政策與帳號方案（企業／教育版有不同的資料保護條款）。機密資料不要用個人帳號上傳。

連上你的工具

Agent 不只整理檔案——還能連上你的信箱、資料夾、行事曆等軟體。

像 Claude Code、Codex 這類 agent，除了讀寫檔案，還能透過連接器或 MCP 連到你常用的工具，把跨軟體的瑣事一起做掉。以下是三類最實用的。

① 整理信件（Gmail / Outlook）

把「收件匣地獄」交給 agent：分類、摘要、抓出待辦、草擬回覆。

給 AI 的指令
請連到我的信箱，幫我處理今天的未讀信：
1) 依主題分類（客戶、供應商、內部、廣告）
2) 每封用一句話摘要，標出需要我回覆或有期限的
3) 把「需回覆」的整理成待辦清單（寄件者、事項、期限）
先不要寄出任何東西，整理好給我看。

② 整理電腦資料夾

檔案散落、命名混亂？讓 agent 依規則重新命名、歸類、找重複檔。

給 AI 的指令
這個資料夾很亂。請幫我：
1) 列出所有檔案，依類型和日期分類
2) 找出重複或命名不一致的檔
3) 提一個「重新命名 + 分資料夾」的規則給我確認，確認後再動手
請先在副本資料夾操作，不要動到原檔。

③ 透過 MCP 串接軟體（如 Google 行事曆）

MCP（Model Context Protocol）就像給 AI 的「萬用插座」——讓 agent 安全地連到外部軟體：Google 行事曆、Slack、Drive、資料庫、GitHub… 不必為每個軟體寫專屬程式。

3) 確認後幫我建立行事曆提醒

Google 行事曆查空檔、排會議、建立提醒

14// 收尾篇

安全與界線

越強大的工具，越要懂得安全地用。這頁請務必記住。

不要外洩機密

客戶 NDA 資料、未公開設計、報價、員工個資，不要貼進公開 AI。先確認公司核可的工具與方案（企業版通常有資料保護條款）。

AI 會自信地犯錯

數字、規格、報價、來源都可能「看起來對、其實錯」（幻覺）。任何對外或重要輸出，一定要人工驗證。

在副本上操作

讓 agent 動檔案時，先複製一份、在測試資料夾裡做。原始資料永遠留底。

你是負責人

AI 是助手不是決策者。最終判斷、簽核、對外發布由人負責。把它當「很快的草稿機」。

一句話原則

能公開講的、能自己驗證的，就放心用；機密的、要負責的，先確認政策、再人工把關。遇到不確定，問主管或 IT，不要先貼了再說。

資料不外洩：實際要做的設定

用 AI 處理公司資料時，外洩風險主要來自三個地方。對照下表把設定做對，就能大幅降低風險。

風險點該怎麼設定／做法
① 你貼進去的內容機密、客戶 NDA、未公開設計、報價、個資 → 練習時用去識別化／假資料（像本教材的虛構檔）；正式資料只用公司核可的工具與帳號。
② 會不會被拿去訓練模型到設定關閉「用我的內容改善模型」：ChatGPT → 設定 / Data controls 關掉 Improve the model；Claude 一般不以 API／商業資料訓練。企業版（Team / Enterprise）預設不訓練，能用就優先用公司帳號。
③ 資料存在哪裡本機型 agent（Claude Code、Codex CLI）檔案留在你電腦，只把必要片段送雲端；純雲端服務則受該服務條款約束。敏感專案優先用「本機型 + 企業方案」。

讓 Agent 動手時，再加這幾道

  • 最小權限：只讓它存取「這個專案資料夾」，不要給整台電腦或整個雲端硬碟。
  • 先計畫、要批准：開啟「核准才執行」模式（plan / approval），會刪除、寄送、覆蓋的動作一律先確認。
  • MCP／連接器只授權必要範圍：例如行事曆只給「讀」不給「刪」；用完可到帳號設定撤銷授權。
  • API key 收好：金鑰不要貼進聊天、不要 commit 進 Git（放 .env 並加入 .gitignore）。
  • 留副本、可回復：在副本資料夾操作，重要資料先備份。
一條紅線

不確定「這份能不能貼／能不能上傳」時，先問主管或 IT，不要先貼了再說——外洩很難收回。
